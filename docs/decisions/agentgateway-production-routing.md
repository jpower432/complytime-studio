# AgentGateway Production Routing

**Status:** Accepted
**Date:** 2026-05-07

## Context

Studio routes A2A traffic through a custom Go reverse proxy (`internal/agents/agents.go`, 200 LOC) that forwards to the kagent controller at `:8083`. This is kagent's dev/CLI pattern — the same endpoint used by `kagent invoke` and the kagent dashboard. The production pattern kagent prescribes is AgentGateway as the A2A ingress, with the controller handling only CRD lifecycle.

Additionally, MCP tool access has no runtime enforcement. Agent pods connect directly to MCP services. The `tools:` allowlist in `agent.yaml` is documentation only — a misconfigured agent can call any tool the server exposes.

KMCP already deploys AgentGateway as a sidecar for every `MCPServer` CRD with `transportType: stdio`. The binary is in the cluster. The kmcp docs explicitly recommend putting AgentGateway in front of MCP servers with the `kagent.dev/discovery=disabled` label.

## Decision

**Deploy AgentGateway as a standalone proxy for all agent (A2A) and tool (MCP) traffic.** The Studio gateway retains a thin reverse proxy for `/a2a/*` to provide same-origin cookie auth and TLS termination before forwarding to AgentGateway. Route MCP through AgentGateway with CEL policies enforcing per-agent tool allowlists.

### Routing Model

- Browser sends `/a2a/*` to the Studio gateway (same-origin, session cookie)
- Studio gateway forwards `/a2a/*` to AgentGateway preserving the full path for HTTPRoute matching
- AgentGateway routes to the correct agent pod based on `PathPrefix /a2a/{agent-id}`
- Agent pods hit AgentGateway for MCP tool calls
- Studio gateway serves CRUD, SPA, and directory API (`/api/*`)
- In direct mode (AgentGateway disabled), the gateway proxies to the agent pod directly, stripping the path prefix
- kagent controller manages workload lifecycle (CRDs) — exits the traffic path

### User Interaction Model

User-directed routing. The workbench agent picker is the orchestrator. No LLM-driven delegation.

### Auth Model

OAuth2 Proxy injects `Authorization: Bearer` header on all traffic. AgentGateway passes headers to backends unchanged. No OIDC config on AgentGateway itself. Upgrade to JWT validation + CEL on claims when per-agent user RBAC is needed.

### Tool Enforcement

CEL `AuthorizationPolicy` on AgentGateway. Rules generated by Helm from `agentDirectory[].tools`. Deny-all default. Agent pods identify themselves via `X-Agent-ID` header on MCP requests.

## Consequences

- Custom A2A proxy code deleted (~200 LOC)
- `KAGENT_A2A_URL`, `KAGENT_AGENT_NAMESPACE`, `A2A_PROXY_URL` removed from gateway
- Workbench A2A endpoint changes from `/api/a2a/` to `/a2a/`
- Agent MCP env vars change from direct service URLs to AgentGateway URLs
- `deploy/kind/setup.sh` installs AgentGateway alongside kagent
- `agentDirectory` in values.yaml becomes the single source of truth for routing, policies, and workbench metadata

## Rejected Alternatives

| Option | Reason |
|:--|:--|
| Keep custom proxy, add tool enforcement in agent code | Honor-system enforcement. Agent can bypass checks. Not auditable. |
| Use kagent controller:8083 as production A2A endpoint | Dev pattern. Not what kagent docs prescribe. Controller not designed for production traffic load. |
| Deploy AgentGateway for A2A only, keep direct MCP | Leaves tool access unenforced. Two patterns for traffic routing. |
| LLM-driven delegation (`a2a_delegate` tool) | Non-deterministic routing. Opaque to user. Not auditable for compliance. |
| Separate subdomain for AgentGateway | CORS complexity for no benefit over path-based ingress split. |
